## 현재 판단

`OBS-001`부터 `OBS-003`까지는 해결됐고, 하루 관찰 체크리스트의 핵심 항목도 모두 통과했습니다. 현재 남은 위험은 기능 오류보다 사용자가 매일 화면을 열었을 때 무엇부터 확인해야 하는지 판단하는 비용입니다.

이미 화면에는 위험도, 시세 품질, 배당 일정, 매도 판단, 알림 상태, 실적 체크일, 실패 사유가 각각 표시됩니다. 하지만 이 정보가 종목 카드와 관리자 패널에 흩어져 있어 종목 수가 늘면 먼저 볼 항목을 사용자가 직접 찾아야 합니다.

## 선정 작업

다음 개발건은 **오늘 확인할 일 카드**입니다.

사용자 화면 `/app`의 포트폴리오 요약 아래에 오늘 먼저 확인할 항목을 모으는 카드 또는 리스트를 추가합니다.

포함 후보:

- 알림 기준까지 가까운 보유 종목
- 이미 알림 상태인 종목
- 시세 또는 배당 조회 실패 종목
- 배당락일, 지급일, 실적 체크일이 임박한 종목
- 보유 중인데 알림이 꺼진 종목
- 매도 판단 기준을 넘었거나 기준가 근처인 종목

## 합격 기준

- 사용자는 화면 첫 진입에서 오늘 확인할 종목과 이유를 5개 이내로 볼 수 있습니다.
- 확인 항목이 없으면 빈 카드 대신 간결한 정상 상태를 보여줍니다.
- 각 항목은 종목명, 이유, 우선순위, 관련 값, 가능한 빠른 행동을 포함합니다.
- 모바일 폭에서도 가로 스크롤 없이 표시됩니다.
- 주문, 자동 매매, 매수/매도 추천 문구는 넣지 않습니다.

## 구현 결과

2026-05-26에 `12.17 오늘 확인할 일 카드`를 구현했습니다. 사용자 화면 `/app`의 포트폴리오 요약 아래에서 알림 기준 도달, 시세 조회 실패, 배당 조회 실패, 기준 근접, 배당락일/지급일 임박, 실적 체크일, 보유 종목 알림 꺼짐을 최대 5개까지 우선순위로 보여줍니다.

각 항목은 종목명, 확인 이유, 세부 값, 빠른 `종목 보기` 동작을 포함합니다. 주문, 자동 매매, 매수/매도 추천 문구는 넣지 않았습니다.

이후 같은 날 `종목 목록 저장 필터`도 구현했습니다. 사용자가 선택한 감시 종목 필터와 정렬 방식은 브라우저 로컬 저장소에 저장되고, 다음에 사용자 화면을 열면 같은 보기 상태로 복원됩니다.

이후 `CSV 가져오기/내보내기`도 구현했습니다. 사용자 화면에서 CSV 양식을 내려받고, 현재 종목 목록을 내보내며, CSV 파일은 전체 행 검증을 통과한 경우에만 기존 `/api/stocks` 등록 API로 일괄 등록합니다. 중복 종목, 숫자/날짜 오류, 알림 기준별 필수값 누락이 있으면 가져오기를 시작하지 않습니다.

이후 `알림 기준 설명 고도화`도 구현했습니다. 등록 팝업과 종목 편집 화면에서 최고가 대비 하락률, 이익금 반납률, 매수가 대비 손절률, 직접 기준가의 필요 입력값, 계산식, 예시 기준가, 주의점을 함께 보여주며, 투자 권유나 매도 추천이 아니라 알림 조건 계산 방식 비교로 한정했습니다.

이후 `배당 API 자동 검증 대시보드 확장`도 구현했습니다. 관리자 운영 진단에서 최근 자동 검증 상태, 다음 예상 확인 시각, 미검증 종목 수, provider별 성공/실패, 필요한 API 키 또는 종목 매칭 점검 조치를 함께 보여줍니다.

이후 `로컬 장중 실사용 재검증`도 구현했습니다. `check:observation --live-session`으로 실제 등록 종목의 시세 최신성, 배당 진단 최신성, 배당 provider 실패, 알림 켜짐/꺼짐/쉬기 상태를 읽기 전용으로 점검합니다.

이후 `로컬 점검 결과 저장/히스토리`도 구현했습니다. `check:observation --save-history`로 점검 결과를 `data/observation-history/`에 저장하고 직전 점검과 실패/수동/통과 개수 및 항목별 상태 변화를 비교합니다.

이후 `로컬 점검 히스토리 관리자 화면 노출`도 구현했습니다. 관리자 화면에서 최근 READY/NOT READY, 실패/수동/통과 개수, 직전 점검 대비 변화, 상태가 바뀐 항목, 최근 점검 파일을 확인합니다.

이후 `관리자 점검 실행/히스토리 저장`도 구현했습니다. 관리자 화면의 `점검 실행/저장` 버튼이 읽기 중심 실사용 점검과 장중 상태 확인을 실행하고, 결과를 `data/observation-history/`에 저장한 뒤 히스토리 목록을 바로 갱신합니다.

이후 `점검 히스토리 상세 보기/다운로드`도 구현했습니다. 관리자 화면에서 저장된 점검 파일의 항목별 evidence와 next action을 열어 보고, 같은 스냅샷을 JSON 파일로 내려받을 수 있습니다.

이후 `점검 히스토리 보관 기간/삭제 관리`도 구현했습니다. 관리자 화면에서 전체 히스토리 파일 수와 기본 보관 개수를 보고, 개별 점검 기록을 삭제하거나 최근 N개만 남기도록 오래된 파일을 정리할 수 있습니다.

다음 후순위 후보는 실패/수동 점검 항목을 보고 실제 조치 내용과 처리 상태를 남기는 `점검 실패 항목 조치 메모`입니다.
