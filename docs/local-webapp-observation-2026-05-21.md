# 로컬 웹앱 실사용 관찰 리포트

날짜 기준: 2026-05-21

목적: 로컬 PC에서 실제 사용자가 Stock Alarm 웹앱을 하루 동안 켜 두고 쓸 때 불편하거나 위험한 지점을 찾기 위한 관찰 기준과 초기 확인 결과를 남깁니다.

## 관찰 범위

이번 관찰은 앱스토어/플레이스토어 준비가 아니라 로컬 웹앱 사용성을 기준으로 봅니다.

- 서버 시작, 상태 확인, 안전 종료
- 사용자 화면 `/app`
- 관리자 화면 `/admin`
- 종목 등록, 보유/관심/매도 상태 변경
- 알림 켜기/끄기, 1시간 쉬기, 오늘 쉬기
- 매도 판단 대시보드
- 백업 미리보기, 복구 전 확인, 자동 백업 상태
- 시세 품질 배지와 provider 실패 사유
- 서버 종료 또는 캐시 화면에서 연결 실패 안내

## 시작 전 안전 확인

테스트 전에 현재 포트와 실행 주체를 먼저 확인합니다.

```powershell
node scripts\local-server.js status
```

실행 중인 서버가 있으면 `data/server.json`의 프로젝트 경로와 `/api/health`가 Stock Alarm인지 확인한 뒤 진행합니다.

운영 데이터로 직접 테스트할 때는 먼저 백업을 만듭니다.

```powershell
node scripts\local-server.js start
```

브라우저에서 관리자 화면에 접속해 `백업 생성`을 누르거나, 텔레그램에서 `/backup`을 실행합니다.

## 2026-05-27 장중 재검증 도구화

OBS-009를 닫기 위해 `check:observation`에 로컬 장중 실사용 재검증용 읽기 전용 `--live-session` 옵션을 추가했습니다. 이 옵션은 실제 등록 종목을 대상으로 장중 시세 최신성, 배당 자동 검증 최신성, 배당 provider 실패 여부, 알림 켜짐/꺼짐/쉬기/알림 진입 상태를 한 번에 점검합니다.

실행 명령:

```powershell
npm run check:observation -- --base-url http://127.0.0.1:3001 --live-session
npm run check:observation -- --base-url http://127.0.0.1:3001 --run-state-check --live-session
npm run check:observation -- --base-url http://127.0.0.1:3001 --live-session --save-history
```

기본 기준은 시세 30분, 배당 진단 72시간입니다. 장중 즉시 확인과 배당 새로고침을 실행한 뒤 `--live-session`을 다시 돌리면 실제 데이터 흐름이 최신 상태인지 확인할 수 있습니다.

## 2026-05-27 점검 히스토리 도구화

OBS-010을 닫기 위해 `check:observation`에 `--save-history` 옵션을 추가했습니다. 이 옵션은 점검 결과를 `data/observation-history/observation-*.json` 파일로 저장하고, 직전 점검 대비 `failed`, `manual`, `passed` 변화와 항목별 상태 변경을 터미널 보고서와 JSON 결과에 함께 표시합니다.

WBS 작업명은 `로컬 점검 결과 저장/히스토리`이며, 이번 범위는 파일 저장과 직전 결과 비교까지입니다.

보관 개수는 기본 30개입니다. 필요하면 `--history-limit 60`으로 늘리거나 `--history-dir data/custom-observation-history`로 저장 위치를 바꿀 수 있습니다. 히스토리 파일은 운영 기록이므로 Git에는 올리지 않습니다.

## 2026-05-27 점검 히스토리 화면화

OBS-011을 닫기 위해 관리자 화면에 `점검 히스토리` 카드를 추가했습니다. `/api/observation-history`가 `data/observation-history/`의 최근 점검 결과를 읽고, 화면에서는 최근 READY/NOT READY, 실패/수동/통과 개수, 직전 점검 대비 변화, 상태가 바뀐 항목, 최근 파일 목록을 확인합니다.

현재 히스토리는 `check:observation --save-history`로 만든 파일을 읽는 방식입니다. 다음 단계에서는 관리자 화면에서 점검 실행과 히스토리 저장까지 바로 처리할 수 있게 합니다.

## 2026-05-27 관리자 점검 실행/저장

OBS-012를 닫기 위해 관리자 화면 `점검 히스토리` 카드에 `점검 실행/저장` 버튼을 추가했습니다. 버튼은 `/api/observation-history/run`을 호출하고, 서버는 현재 실행 중인 로컬 주소를 기준으로 읽기 중심 실사용 점검과 장중 상태 확인을 실행한 뒤 결과를 `data/observation-history/`에 저장합니다.

버튼 기본값은 안전을 위해 검증용 종목을 생성/삭제하는 상태 변경 점검을 포함하지 않습니다. 실행 직후 화면에는 방금 실행한 READY/NOT READY, 실패/수동/통과 개수, 확인이 필요한 항목, 저장 파일명이 표시되고, 아래 히스토리 목록도 바로 갱신됩니다.

## 2026-05-28 점검 히스토리 상세/다운로드

OBS-013을 닫기 위해 관리자 화면 `점검 히스토리` 목록에 `상세`와 `JSON` 버튼을 추가했습니다. `상세`는 `/api/observation-history/{fileName}`을 호출해 저장된 점검 파일의 실패/수동 항목 evidence와 next action을 화면에서 바로 표시합니다.

`JSON` 또는 `JSON 다운로드`는 같은 상세 응답의 저장 스냅샷을 브라우저에서 내려받습니다. 히스토리 파일명은 `observation-*.json` 형식만 허용해 다른 로컬 파일을 열지 못하게 막았습니다.

## 2026-05-29 점검 히스토리 보관 관리

OBS-014를 닫기 위해 관리자 화면 `점검 히스토리` 카드에 보관 관리 영역과 개별 삭제 버튼을 추가했습니다. 각 히스토리 행의 `삭제`는 해당 `observation-*.json` 파일만 삭제하고, 삭제 전 브라우저 확인 메시지를 띄웁니다.

보관 관리 영역에서는 전체 히스토리 파일 수와 기본 보관 개수를 표시하고, `오래된 기록 정리`로 최근 N개만 남길 수 있습니다. 서버 API는 `DELETE /api/observation-history/{fileName}`와 `POST /api/observation-history/prune`이며, 파일명 검증과 관리자 토큰 보호를 적용합니다.

## 2026-05-29 점검 실패 항목 조치 메모

OBS-015를 닫기 위해 관리자 화면 `점검 히스토리` 상세의 실패/수동 항목에 조치 상태, 조치 메모, 다음 확인일 입력 영역을 추가했습니다. 상태는 `미처리`, `조치중`, `해결`, `보류`로 관리하고, 저장된 메모는 상세 화면과 JSON 다운로드 스냅샷에 함께 표시됩니다.

히스토리 원본 파일은 점검 당시 스냅샷으로 유지하고, 조치 메모는 `data/observation-actions.json`에 별도 저장합니다. 서버 API는 `PATCH /api/observation-history/{fileName}/results/{resultId}/action`이며, 파일명과 항목 ID 검증 및 관리자 토큰 보호를 적용합니다.

| OBS-003 | 낮음 | 즉시 확인과 알림 제어의 실제 장중 저장 흐름은 자동 smoke check에서 실행하지 않음 | 해결 | `--run-state-check`로 검증용 종목 생성, 즉시 확인, 알림 제어 저장, 삭제 흐름 확인 |
| OBS-004 | 중간 | 위험 신호, 실패 사유, 배당/실적 일정, 알림 상태가 종목 카드에 흩어져 오늘 먼저 볼 항목을 직접 찾아야 함 | 해결 | 사용자 화면에 `오늘 확인할 일 카드`를 추가 |
| OBS-005 | 낮음 | 종목 필터와 정렬 선택이 방문마다 초기화되어 반복 조회 시 같은 보기 상태를 다시 맞춰야 함 | 해결 | `종목 목록 저장 필터`를 구현 |
| OBS-006 | 낮음 | 종목 수가 늘면 하나씩 등록하거나 다른 도구의 보유 목록을 옮기는 과정이 번거로움 | 해결 | `CSV 가져오기/내보내기`를 구현 |
| OBS-007 | 낮음 | 알림 기준 종류가 늘어 처음 쓰는 사용자가 기준별 필요 입력값과 계산 차이를 비교하기 어려움 | 해결 | `알림 기준 설명 고도화`를 구현 |
| OBS-008 | 낮음 | 배당 API 자동 갱신 결과가 provider 진단 중심이라 최근 자동 검증 상태와 다음 조치를 빠르게 파악하기 어려움 | 해결 | `배당 API 자동 검증 대시보드 확장`을 구현 |
| OBS-009 | 중간 | 사용자 편의 개선과 배당/시세 진단 기능이 누적되어 실제 장중 데이터로 전체 흐름을 다시 확인할 필요가 있음 | 해결 | `check:observation --live-session` 장중 재검증 옵션을 추가 |
| OBS-010 | 낮음 | 장중 재검증 결과가 현재 터미널 출력에만 남아 이전 점검과 비교하기 어려움 | 해결 | `check:observation --save-history` 파일 히스토리와 직전 결과 비교를 추가 |
| OBS-011 | 낮음 | 저장된 점검 히스토리를 관리자 화면에서 아직 직접 볼 수 없어 파일을 열어 확인해야 함 | 해결 | `로컬 점검 히스토리 관리자 화면 노출`을 구현 |
| OBS-012 | 중간 | 점검 히스토리를 만들려면 아직 터미널에서 `check:observation --save-history`를 실행해야 함 | 해결 | `관리자 점검 실행/히스토리 저장`을 구현 |
| OBS-013 | 낮음 | 저장된 과거 점검의 세부 evidence와 다음 조치를 보려면 아직 JSON 파일을 직접 열어야 함 | 해결 | `점검 히스토리 상세 보기/다운로드`를 구현 |
| OBS-014 | 낮음 | 점검 히스토리가 계속 쌓이면 오래된 파일을 관리자가 화면에서 정리하기 어려움 | 해결 | `점검 히스토리 보관 기간/삭제 관리`를 구현 |
| OBS-015 | 낮음 | 실패/수동 점검 항목을 보고 실제 어떤 조치를 했는지 별도 메모로 남길 곳이 없음 | 해결 | `점검 실패 항목 조치 메모`를 구현 |

## 다음 개발 연결

관찰 리포트의 초기 기준은 준비됐습니다. 실제 사용 중 발견되는 이슈는 이 문서에 `OBS-*`로 추가하고, 개발이 필요한 항목은 WBS에 별도 작업으로 승격합니다.

현재 다음 개발 후보는 앱 개발 재개 시 모바일 실기기 E2E 테스트입니다.

- 실제 장중 사용 중 나온 불편을 `OBS-*`로 추가
- 재현 가능한 이슈는 WBS 작업으로 승격
- 앱 관련 작업은 사용자가 재개하기 전까지 보류 항목으로 유지
- 관리자 화면에서 `실사용 이슈` 카드를 보고 열린 이슈와 우선 처리 대상을 확인
